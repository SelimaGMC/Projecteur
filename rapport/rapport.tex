    \chapter{Introduction}
 
    Ce projet consiste à implémenter un système RAG (\textit{Retrieval-Augmented Generation}) sur le sujet de notre choix. Nous avons décidé de nous pencher sur le cinéma, et plus particulièrement le cinéma français, en nous appuyant sur la liste Wikipedia des plus grands succès français au box-office mondial (voir annexe), ainsi que sur les pages Wikipedia dédiées à chacun de ces films.

    Pour le modèle de langage, nous avons opté pour \textbf{Ollama} afin de pouvoir faire tourner le système entièrement en local, sans dépendance à une API externe. Nous avons suivi le notebook officiel de Gemini (voir annexe) comme base de départ, puis nous avons adapté et enrichi le code pour notre sujet.

    \section{Architecture générale}
 
    Nous avons divisé le code en plusieurs fichiers Python afin de faciliter la compréhension, la maintenance et les tests. Notre structure a évolué au fil du projet et des implémentations conservées. Le système final suit l'arborescence suivante :
 
    \begin{verbatim}
    [main.py]
 |
 +---> [crawler.py]      -> Récupération des URLs des films
 |
 +---> [retriever.py]    -> Construction de la base vectorielle (Chroma)
 |         |
 |         +---> [film_parser.py] -> Extraction du contenu des pages Wikipedia
 |
 +---> [generator.py]    -> Génération des réponses avec Ollama
           |
           +---> [router.py] -> 

 +---> [app.py] -> Application Streamlit
    \end{verbatim}
    
    \chapter{Description des modules}
 
    \section{Module principal (main.py)}
 
    Ce fichier est le point d'entrée du programme. C'est lui qui gère l'exécution des autres modules dans l'ordre, correspondant aux étapes suivantes :
 
    \begin{itemize}
        \item \textbf{Configuration :} on initialise les paramètres du système (modèle, agent utilisateur).
        \item \textbf{Crawling :} on récupère les URLs des pages Wikipedia de chacun des films via la page initiale qui les liste.
        \item \textbf{Indexation :} on construit la base vectorielle à partir de ces pages. Si elle existe déjà sur le disque, elle est chargée directement pour éviter de tout recalculer.
        \item \textbf{Interaction :} correspond à la boucle de questions/réponses entre le système et l'utilisateur. L'utilisateur pose une question, et le système affiche la réponse générée par le LLM. Cette boucle continue jusqu'à ce que l'utilisateur l'interrompe (Ctrl+C).
    \end{itemize}
 
    \section{Module de crawler (crawler.py)}
 
    Ce fichier comporte deux fonctions :
 
    \begin{itemize}
        \item \textbf{get\_movie\_titles() :} cette première fonction extrait le titre et l'année de sortie des films depuis la page Wikipedia initiale. Pour cela, elle utilise la librairie BeautifulSoup pour parser le contenu HTML, puis cible les balises \verb|<table class='wikitable'>| et en récupère la deuxième colonne (titre) ainsi que la première colonne (année) de chaque ligne. Elle nettoie ensuite les références de type \texttt{[1]} inutiles avec un regex.

        \item \textbf{load\_movie\_urls() :} cette seconde fonction convertit chaque titre en URL Wikipedia via l'API Wikipedia. Elle gère plusieurs cas spécifiques pour éviter les erreurs. En cas d'homonymie \\ (\texttt{DisambiguationError}), le système tente jusqu'à trois variantes du titre dans l'ordre suivant : d'abord le titre exact, puis \texttt{titre (film)}, et enfin \texttt{titre (film, année)} pour les pages dont le nom inclut l'année de sortie (comme \textit{Jeanne d'Arc (film, 1999)}). La fonction gère également les erreurs réseau en attendant quelques secondes avant de continuer, pour respecter les limites de l'API Wikipedia.
    \end{itemize}
 
    Plusieurs choix d'implémentation permettent le bon fonctionnement de ce module. L'utilisation d'un dictionnaire pour dédoublonner les titres garantit qu'on ne traite pas deux fois le même film, même s'il apparaît dans les deux tableaux de la page. Le paramètre \texttt{auto\_suggest=False} évite les suggestions automatiques de Wikipedia qui pourraient mener à des pages non souhaitées.

    \section{Module d'extraction (film\_parser.py)}
 
    Ce fichier est le c\oe{}ur de l'extraction des données. C'est lui qui permet de rendre la base de connaissances suffisamment complète pour répondre aux questions de l'utilisateur.
 
    Contrairement au notebook qui automatise ce processus avec \texttt{UnstructuredURLLoader}, nous avons fait un parsing personnalisé adapté au format des pages Wikipedia. On analyse la page de chaque film récupérée avec le module précédent, et deux sorties distinctes sont produites selon le type de données. L'élément renvoyé suit le format \texttt{tuple[str, dict]} :
 
    \begin{itemize}
        \item \textbf{Le str} contient un texte continu destiné à la recherche vectorielle. Il comprend l'introduction si elle existe, ainsi que les sections synopsis, accueil et analyse.
        \item \textbf{Le dictionnaire} contient des données structurées destinées à être indexées séparément dans Chroma : la fiche technique, la distribution et les distinctions.
    \end{itemize}
 
    Nous avons fait ce choix de séparation afin d'avoir le maximum d'informations utiles pour répondre aux questions, tout en évitant de noyer des données factuelles précises (comme un réalisateur ou un nombre de César) dans des passages narratifs.

    Ces éléments sont extraits grâce à BeautifulSoup en parcourant les balises \texttt{<h2>} pour organiser le contenu en sections, puis le texte des paragraphes \texttt{<p>} et des listes \texttt{<ul>} ou \texttt{<ol>}. Pour les sections structurées, on stocke temporairement les éléments HTML bruts dans \texttt{sql\_elems} pour les traiter séparément. Les références Wikipedia et les espaces multiples sont nettoyés par la fonction \texttt{clean\_wiki\_refs()}.

    Les distinctions sont organisées par sous-catégories (par exemple \texttt{[César]}, \texttt{[BAFTA]}) grâce aux balises \texttt{h3/h4}, ce qui permet de conserver le contexte de chaque récompense dans une liste plate.

    Ces fonctions sont compatibles avec les deux formats de Wikipedia : l'ancien format avec balises \texttt{<h2>} directes, et le nouveau format depuis 2023 avec des \verb|<div class="mw-heading2">|.
 
    \section{Module de base de données SQL (database.py)}
 
    Nous avons initialement tenté de mettre en place une BDD Sqlite3 contenant les informations exactes (fiche technique, distribution, etc.) répartis lors du module d'extraction dans des tables structurées.
    On attendrait donc du générateur qu'il puisse en premier lieu estimer si la question nécessite une réponse trouvée dans la base de données, dans le dossier chroma\_db, ou les deux, avant de formuler une requête SQL si nécessaire (le modèle des tables lui étant fourni) et enfin de récupérer la ou les information(s) désirée(s).

    Cette fonctionnalité a été initialement mise en place comme décrit ci-haut, avant d'être retirée pour des soucis de performance. En effet, le choix du LLM employé ne s'apprêtait pas à ce type de requêtes, malgré les différentes tentatives d'amélioration et d'accomodation.
    Suite à cela, les informations exactes ont été traitées comme nous avons pu le voir lors de la description des modules.
    
    \section{Module de retriever (retriever.py)}
 
    Ce fichier construit la base vectorielle qui permet la recherche sémantique. Il suit les étapes suivantes :
 
    \begin{itemize}
        \item \textbf{Vérification de persistance :} si Chroma existe déjà sur le disque et n'est pas vide, on le charge directement, évitant ainsi de tout recalculer à chaque lancement.
        \item \textbf{Extraction :} pour chaque URL, on extrait le texte et les données structurées avec \texttt{film\_parser.py}.
        \item \textbf{Découpage (\textit{chunking}) :} le texte narratif est découpé en segments de 1000 caractères avec un chevauchement de 200 caractères, pour ne pas couper les informations en plein milieu d'une phrase.
        \item \textbf{Contextual Retrieval :} chaque chunk est préfixé par \texttt{[Film : <titre>]} avant indexation, selon le principe du \textit{Contextual Retrieval}\footnote{\url{https://www.anthropic.com/news/contextual-retrieval}} proposé par Anthropic (2024). Cela permet au LLM d'identifier à quel film se rapporte un extrait même sorti de son contexte, et évite les confusions entre films lors de la génération (voir tableau~\ref{tab:conception}).
        \item \textbf{Indexation des données structurées :} la fiche technique, la distribution et les distinctions sont converties en documents textuels séparés et indexées avec des métadonnées de type (\texttt{fiche\_technique}, \texttt{distribution}, \texttt{distinctions}). Ces métadonnées sont ensuite exploitées par le routeur pour filtrer les chunks selon la nature de la question.
        \item \textbf{Calcul des embeddings et stockage :} les embeddings sont calculés en parallèle par lots de 50 chunks avec 4 threads, puis stockés dans Chroma.
    \end{itemize}

    Pour les embeddings, nous avons choisi le modèle \textbf{BGE-M3} via Ollama. C'est un modèle multilingue qui offre un meilleur support du français que le modèle \texttt{nomic-embed-text} utilisé initialement, avec une dimension de vecteur de 1024 (contre 768). Ce changement nécessite de reconstruire la base Chroma car les dimensions des vecteurs sont incompatibles.

    \section{Module de génération (generator.py)}

    Ce fichier définit la chaîne RAG et les paramètres du modèle de génération.

    \subsection{Modèle de génération}

    Nous utilisons \textbf{Qwen2.5 7B} (\texttt{qwen2.5:7b}) via Ollama comme modèle de génération. Ce modèle offre un bon équilibre entre qualité de réponse et rapidité d'exécution en local. La fenêtre de contexte est fixée à 8192 tokens, ce qui est suffisant pour accueillir les 8 chunks retournés par le retriever (environ 2000--2500 tokens) ainsi que le prompt système.

    \subsection{Prompt système}

    Le prompt système définit les règles strictes que le modèle doit suivre :
    \begin{itemize}
        \item Répondre uniquement à partir des extraits fournis, sans utiliser de connaissances générales.
        \item En l'absence d'information, répondre explicitement \textit{«~Je ne dispose pas de cette information dans mes sources.~»}
        \item Ne pas mélanger les informations de films différents dans une même affirmation, sauf si la question demande explicitement une comparaison.
        \item Préciser le titre du film concerné lorsque plusieurs films apparaissent dans les extraits.
    \end{itemize}

    \subsection{Adaptive RAG}

    Nous avons enrichi le module de génération avec un mécanisme d'\textbf{Adaptive RAG} : plutôt que d'utiliser systématiquement les mêmes paramètres de retrieval, le système adapte sa stratégie de recherche selon la nature de la question posée. Cette logique est implémentée dans un fichier \texttt{router.py} dédié.

    Le routeur classe chaque question dans l'une des trois catégories suivantes à l'aide de règles déterministes basées sur des mots-clés :

    \begin{itemize}
        \item \textbf{Factuelle} (ex. \textit{«~Qui a réalisé Intouchables~?~»}) : le retrieval se limite à $k=4$ chunks, filtrés sur les types \texttt{fiche\_technique}, \texttt{distribution} et \texttt{distinctions}. On évite ainsi de ramener des passages narratifs inutiles pour une question factuelle.
        \item \textbf{Narrative} (ex. \textit{«~De quoi parle Les Visiteurs~?~»}) : comportement par défaut avec $k=8$ chunks sur l'ensemble du corpus, sans filtre.
        \item \textbf{Comparaison} (ex. \textit{«~Compare Intouchables et Bienvenue chez les Ch'tis~»}) : on augmente à $k=14$ chunks avec une recherche par \textit{Maximum Marginal Relevance} (MMR) pour maximiser la diversité des résultats et s'assurer que les deux films sont bien représentés dans le contexte envoyé au LLM.
    \end{itemize}

    Ce mécanisme s'intègre dans la chaîne LangChain via un \texttt{RunnableLambda} qui intercepte la question avant le retrieval, choisit la stratégie adaptée, puis passe la main au reste de la chaîne sans modifier le prompt ni le modèle.

    \section{Application Streamlit (app.py)}
    Ce module fournit une interface graphique simple et moderne, baptisée «~Pro'jecteur~», permettant de substituer la boucle d'interaction du main par une expérience utilisateur web fluide.
    L'application s'articule autour de plusieurs mécanismes clés :

    \begin{itemize}
        \item \textbf{Mise en cache :}  Pour éviter de reconstruire la chaîne RAG et de recharger les structures de données lourdes (ChromaDB) à chaque message, le décorateur \texttt{st.cache\_resource} est appliqué à la fonction d'initialisation.  De plus, un fichier de cache local nommé \texttt{movies\_list.json} permet à l'utilisateur de connaître les films connus par le RAG.

        \item \textbf{Gestionnaire de l'historique :} L'état de la discussion est maintenu à court terme dans la mémoire de la session via le dictionnaire \texttt{st.session\_state}. 
        Pour assurer la pérennité des échanges, un système de sérialisation sur disque a été mis en place dans un fichier \texttt{conv\_history.json}. L'utilisateur a donc accès à un historique fonctionnel sur lequel il peut recharger une ancienne session ou basculer d'une discussion à l'autre.

        \item \textbf{Suggestions de prompts :} L'interface intègre deux dispositifs complémentaires d'aide à la formulation :
        \begin{itemize}
            \item \textit{Boutons de suggestions :} Placés sur la page d'accueil, ils affichent des questions types pré-définies (ex. \textit{«~Qui a réalisé Intouchables~?~»}) qui s'injectent directement dans le pipeline de traitement via une variable d'état transitoire \texttt{pending\_q}.
            \item \textit{Explorateur dynamique de films :} Un menu déroulant (\texttt{st.sidebar.expander}) liste dynamiquement tous les films indexés. Chaque titre de film est encapsulé dans un bouton stylisé cliquable, qui permet de formuler une requête sur le film en question (ex. \textit{«~Parle-moi du film [Titre]~»}).
        \end{itemize}
    \end{itemize}

    \chapter{Évaluation}
    \label{sec:eval}

    Nous avons mis en place un module d'évaluation indépendant (\texttt{eval\_qa.py}) pour mesurer les performances du système de manière reproductible. Ce module permet de tester le retriever et la chaîne RAG séparément, et de comparer l'impact des différentes améliorations apportées au fil du projet.

    \section{Métrologie}

    L'évaluation du retriever repose sur deux métriques calculées sur un jeu de questions de référence :
    \begin{itemize}
        \item \textbf{Précision :} proportion de chunks retournés qui sont effectivement pertinents pour la question.
        \item \textbf{Recall :} proportion de chunks pertinents effectivement retrouvés parmi tous les chunks pertinents existants.
    \end{itemize}

    Pour la qualité des réponses générées, nous utilisons une évaluation automatique : le LLM est invité à noter la réponse produite par rapport à une réponse de référence, selon plusieurs critères (exactitude, complétude, absence d'hallucination).

    \section{Résultats}

    Les améliorations successives apportées au projet (passage de \texttt{nomic-embed-text} à \texttt{bge-m3}, ajout du Contextual Retrieval, implémentation de l'Adaptive RAG) ont été mesurées avec ce module pour valider leur impact sur les performances. Ces résultats sont disponibles dans le fichier \texttt{eval\_results\_run.json}.

    \chapter{Conclusion}

    Ce projet nous a permis de construire un système RAG complet, en allant au-delà du notebook de départ. Nous avons notamment implémenté un parsing personnalisé adapté au format des pages Wikipedia, un indexage différencié entre données narratives et données structurées, et un mécanisme d'Adaptive RAG pour adapter dynamiquement la stratégie de retrieval selon la nature de la question. \\
    
    Une piste d'indexation des données structurées sous forme d'une base de données SQL a été envisagée, bien que finalement abandonnée faute de résultats satisfaisants. \\

    On remarque à travers ce projet que la qualité d'un RAG repose en grande partie sur la qualité de l'indexation : un bon découpage, des métadonnées bien choisies et un modèle d'embedding adapté à la langue ont un impact beaucoup plus fort sur les résultats que le choix du modèle de génération lui-même.

    Parmi les pistes d'amélioration envisageables, on peut citer l'ajout d'un hybrid search (BM25 + vecteur) pour mieux retrouver les noms propres, un mécanisme de query rewriting pour reformuler les questions avant le retrieval, ou encore un reranking par cross-encoder pour affiner l'ordre des chunks retournés.
